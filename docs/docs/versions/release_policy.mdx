# LangChain 发布策略

LangChain 生态系统由不同的组件包组成（例如 `langchain-core`、`langchain`、`langchain-community`、`langgraph`、`langserve`、合作伙伴包等）。

## 版本控制

### `langchain`、`langchain-core` 和集成包

`langchain`、`langchain-core`、`langchain-text-splitters` 和集成包（`langchain-openai`、`langchain-anthropic` 等）遵循 [语义化版本](https://semver.org/) 格式 0.**Y**.**Z**。这些包正处于快速开发阶段，因此目前版本号的主版本号为 0。

次要版本号的增加将发生在：

- 对任何未标记为 `beta` 的公共接口进行破坏性更改时。

补丁版本号的增加将发生在：

- 错误修复，
- 新功能，
- 对私有接口的任何更改，
- 对 `beta` 功能的任何更改。

在次要版本之间升级时，用户应查看破坏性更改和弃用列表。

我们将不时将包版本化为 **发布候选版**。这些版本旨在作为稳定版本发布，但我们希望在发布前获得社区的反馈。发布候选版的版本号将为 0.**Y**.**Z**rc**N**。例如，0.2.0rc1。如果没有发现问题，发布候选版将以相同的版本号发布为稳定版本。如果发现问题，我们将发布一个新的发布候选版，并增加 `N` 值（例如，0.2.0rc2）。

### `langchain-community`

次要版本号的增加将发生在：

- 更新必需的 `langchain-x` 依赖项的主/次版本号。例如，当将 `langchain-core` 的必需版本从 `^0.2.x` 更新到 `0.3.0` 时。

补丁版本号的增加将发生在：

- 错误修复，
- 新功能，
- 对私有接口的任何更改，
- 对 `beta` 功能的任何更改，
- 集成的破坏性更改，以反映第三方服务的破坏性更改。

在可能的情况下，我们将避免在补丁版本中进行破坏性更改。
但是，如果外部 API 进行了破坏性更改，则相应的 `langchain-community` 集成可能会在补丁版本中发生破坏性更改。

### `langchain-experimental`

所有更改都将伴随补丁版本号的增加。

## 发布周期

我们预计将 `langchain` 和 `langchain-core` 的 **次要** 版本（例如，从 0.2.x 到 0.3.0）间隔至少 2-3 个月，因为这些版本可能包含破坏性更改。

补丁版本会频繁发布，每周最多几次，因为它们包含错误修复和新功能。

## API 稳定性

LLM 应用的开发是一个快速发展的领域，我们不断从用户和社区中学习。因此，我们预计 `langchain` 和 `langchain-core` 中的 API 将继续演进，以更好地满足用户的需求。

尽管 `langchain` 和 `langchain-core` 目前都处于 1.0 之前的状态，但我们致力于在这些包中维护 API 稳定性。

- 对公共 API 的破坏性更改将导致次要版本号的增加（第二个数字）。
- 任何错误修复或新功能都将导致补丁版本号的增加（第三个数字）。

我们通常会尽量避免不必要的更改，并将为将被删除的功能提供弃用策略。

### 其他包的稳定性

LangChain 生态系统中其他包的稳定性可能会有所不同：

- `langchain-community` 是一个社区维护的包，其中包含第三方集成。虽然我们尽最大努力审查和测试 `langchain-community` 中的更改，但由于其中包含许多社区贡献，`langchain-community` 预计会比 `langchain` 和 `langchain-core` 经历更多的破坏性更改。
- 合作伙伴包可能遵循不同的稳定性和版本控制策略，用户应参考这些包的文档以获取更多信息；但是，总的来说，这些包预计是稳定的。

### 什么是“API 稳定性”？

API 稳定性意味着：

- 所有公共 API（本文档中的所有内容）都不会在未提供向后兼容的别名的情况下被移动或重命名。
- 如果向这些 API 添加新功能（这是很有可能的），它们不会破坏或更改现有方法的含义。换句话说，“稳定”并不（一定）意味着“完整”。
- 如果出于某种原因，必须删除或替换已声明为稳定的 API，它将被声明为已弃用，但将在至少两个次要版本中保留在 API 中。调用已弃用的方法时会发出警告。

### **标记为内部的 API**

某些 API 通过几种方式明确标记为“内部”：

- 一些文档引用了内部内容并将其标记为内部。如果文档说明某项内容是内部的，那么它可能会发生更改。
- 以前导下划线（**`_`**）为前缀的函数、方法和其他对象。这是指示某项内容为私有的标准 Python 约定；如果任何方法以单个 **`_`** 开头，则它是内部 API。
    - **例外：** 某些方法以 `_` 开头，但不包含实现。这些方法旨在被提供实现的子类覆盖。这些方法通常是 LangChain **公共 API** 的一部分。

## 弃用策略

我们通常会避免弃用功能，直到有更好的替代方案可用。

当某个功能被弃用时，它将在 `langchain` 和 `langchain-core` 的当前和下一个次要版本中继续工作。之后，该功能将被删除。

由于我们预计将次要版本间隔至少 2-3 个月，这意味着某个功能可能在被弃用 2-6 个月后被删除。

在某些情况下，如果该功能没有在包中引起问题，我们可能会允许该功能在代码库中保留更长时间，以减轻用户负担。
