# 审查流程

## 概述

本文档概述了 LangChain 维护者用于审查拉取请求 (PR) 的流程。此流程的主要目标是增强 LangChain 的开发者体验。

## 审查状态

我们使用三种主要状态对 PR 进行分类，这些状态在右侧边栏中标记为项目项状态，并可在[此处](https://github.com/orgs/langchain-ai/projects/12/views/1)详细查看。

- **待分类 (Triage)**：
  - 所有新提交 PR 的初始状态。
  - 需要维护者将其归类到其他状态之一。

- **需要支持 (Needs Support)**：
  - 需要社区反馈或额外输入才能继续进行的 PR。
  - 如果获得 5 个赞，将自动晋升到待办事项列表。
  - 应用此状态时会生成自动评论，解释流程和点赞要求。
  - 如果 PR 在此状态下停留 25 天，将通过自动评论标记为“过时”。
  - 如果 30 天内未采取进一步行动，PR 将被自动关闭。

- **审查中 (In Review)**：
  - 我方团队正在积极审查的 PR。
  - 这些 PR 会被定期审查和监控。

**注意**：一个 PR 一次只能有一个状态。

**注意**：您可能会注意到另外三种状态：完成 (Done)、已关闭 (Closed) 和内部 (Internal)，它们
独立于此生命周期。完成和已关闭的 PR 分别表示已合并或已关闭。内部状态适用于核心维护者提交的 PR，这些 PR 由提交者负责。

## 审查指南

1. **触及 /libs/core 的 PR**：
   - 直接影响核心代码并可能影响最终用户的 PR。
   - **分类指南**：大多数 PR 应直接进入 `审查中 (In Review)` 或关闭。
   - 这些 PR 具有最高优先级，审查速度最快。
   - 没有（在 PR 摘要或链接的 issue 中）**简洁**描述其动机的 PR 可能会在未深入审查的情况下被关闭。请勿使用 LLM 生成冗长的 PR 描述。
   - 没有单元测试的 PR 可能会被关闭。
   - 功能请求应首先作为 GitHub issue 打开，并与 LangChain 维护者讨论。未经事先讨论提交的大型 PR 可能会被关闭。

2. **触及 /libs/langchain 的 PR**：
   - 高影响力 PR，与核心 PR 非常相关，但优先级略低。
   - **分类指南**：大多数 PR 应直接进入 `审查中 (In Review)` 或关闭。
   - 这些 PR 的审查和关闭速度与核心 PR 类似。
   - 新功能请求应事先在 issue 中与核心维护者团队讨论。

3. **触及 /libs/partners/ 的 PR**：
   - 涉及集成包的 PR。
   - **分类指南**：大多数 PR 应直接进入 `审查中 (In Review)` 或关闭。
   - 审查可能由我们的团队进行，或根据 PR 的内容移交给合作伙伴的开发团队。
   - 我们与大多数合作伙伴开发团队保持沟通渠道，以促进此流程。

4. **社区 PR**：
   - 大多数社区 PR 将获得“需要支持 (needs support)”的初始状态。
   - **分类指南**：大多数 PR 应进入 `需要支持 (Needs support)`。高流量集成上的 Bug 修复应直接进入 `审查中 (In review)`。
   - **分类指南**：所有新功能和集成都应进入 `需要支持 (Needs support)`，如果它们未获得足够支持（通过点赞或评论衡量），将被关闭。
   - 在 `需要支持 (Needs Support)` 状态下停留 20 天的 PR 将被标记为“过时”，如果 30 天内未采取行动，将被关闭。

5. **文档 PR**：
   - 触及 docs/docs 中文档内容的 PR。
   - **分类指南**：
      - 修复单个文件中拼写错误或小错误的 PR，且 CI 通过的，应直接进入 `审查中 (In Review)`。
      - 在 issue 中已讨论并同意的更改的 PR，应直接进入 `审查中 (In Review)`。
      - 添加新页面或更改文档结构的 PR 应进入 `需要支持 (Needs Support)`。
   - 我们努力标准化文档格式以简化审查流程。
   - CI 作业针对文档运行，以确保符合标准，从而自动化了大部分审查工作。

6. **PR 必须使用英文**：
   - 非英文 PR 将在未审查的情况下被关闭。
   - 这是为了确保所有维护者都能有效地审查 PR。

## 审查优先级

我们的目标是通过专注于构建以下软件来提供最佳的开发体验：

- 可用：按预期工作（无错误）。
- 有用：通过现成的组件和简化应用构建的运行时来改进 LLM 应用开发。
- 简单：易于使用且文档齐全。

我们相信此流程反映了我们的优先事项，如果您认为它不符合，我们欢迎反馈。
