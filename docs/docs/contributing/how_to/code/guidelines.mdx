# 通用指南

在所有类型的贡献中，请牢记以下几点：

- 遵循 ["fork and pull request"](https://docs.github.com/en/get-started/exploring-projects-on-github/contributing-to-a-project) 工作流程。
- 打开 pull request 时，请填写已提交的 pull request 模板。注意相关的 issue 并标记相关的维护者。
- 在请求审查之前，请确保您的 PR 通过了格式、linting 和测试检查。
  - 如果您希望对当前的进度获得评论或反馈，请打开一个 issue 或 discussion 并标记维护者。
  - 有关如何在本地运行这些检查的信息，请参阅 [Testing](setup.mdx#testing) 和 [Formatting and Linting](setup.mdx#formatting-and-linting) 部分。
- 向后兼容是关键。您的更改不得破坏现有功能，除非是关键的错误修复和安全修复。
- 在打开新的 PR 或 issue 之前，请查找是否已有重复的 PR 或 issue 被打开。
- 保持范围尽可能隔离。通常情况下，您的更改一次不应影响多个包。

## Bug修复

我们鼓励并感谢 Bug 修复。我们要求您：

- 详细解释 Bug，以便维护者能够重现它。
  - 如果有相关的 issue，请链接到它。在链接前加上 `Fixes`，这样当 PR 被合并时，该 issue 将自动关闭。
- 尽可能避免破坏性更改。
- 包含在没有 Bug 修复的情况下会失败的单元测试。

如果您遇到一个 Bug 但不知道如何修复，我们要求您为此打开一个 issue，详细描述您遇到 Bug 的环境。

## 新功能

我们的目标是为新功能设定高标准。我们通常不接受来自外部贡献者的新核心抽象、基础设施更改、依赖项更改或新的代理/链，除非有现有的 GitHub discussion 或 issue 证明其有迫切需求。

- 新功能必须附带文档、单元测试，以及（如果适用）集成测试。
- 新集成必须附带文档、单元测试，以及（如果适用）集成测试。
  - 有关贡献新集成的更多详细信息，请参阅 [此页面](../integrations/index.mdx)。
- 新功能不应继承或使用已弃用的方法或类。
- 我们将拒绝可能导致安全漏洞或报告的功能。
- 请勿添加任何硬性依赖项。集成可以添加可选依赖项。
