# 如何实现集成包

本指南将引导您完成 LangChain 集成包的实现过程。

集成包只是可以用 `pip install <your-package>` 安装的 Python 包，其中包含与 LangChain 核心接口兼容的类。

我们将涵盖：

1. （可选）如何引导新的集成包
2. 如何实现符合 LangChain 接口的组件，例如 [聊天模型](/docs/concepts/chat_models/) 和 [向量存储](/docs/concepts/vectorstores/)；

## （可选）引导新的集成包

在本节中，我们将概述引导新集成包的两种选项，如果您愿意，也可以使用其他工具！

1. **langchain-cli**：这是一个命令行工具，可用于引导新的集成包，其中包含 LangChain 组件的模板和用于依赖管理的 Poetry。
2. **Poetry**：这是一个 Python 依赖管理工具，可用于引导新的 Python 包及其依赖项。然后，您可以将 LangChain 组件添加到此包中。

<details>
    <summary>选项 1：langchain-cli（推荐）</summary>

在本指南中，我们将使用 `langchain-cli` 从模板创建新的集成包，该模板可以进行编辑以实现您的 LangChain 组件。

### **先决条件**

- [GitHub](https://github.com) 账户
- [PyPi](https://pypi.org/) 账户

### 使用 langchain-cli 引导新的 Python 包

首先，安装 `langchain-cli` 和 `poetry`：

```bash
pip install langchain-cli poetry
```

接下来，为您的包想一个名字。在本指南中，我们将使用 `langchain-parrot-link`。您可以通过在 [PyPi 网站](https://pypi.org/)上搜索来确认该名称在 PyPi 上可用。

接下来，使用 `langchain-cli` 创建您的新 Python 包，并使用 `cd` 进入新目录：

```bash
langchain-cli integration new

> 要创建的集成名称（例如 `my-integration`）：parrot-link
> 集成名称（PascalCase）[ParrotLink]：

cd parrot-link
```

接下来，让我们添加所需的任何依赖项

```bash
poetry add my-integration-sdk
```

我们还可以将一些 `typing` 或 `test` 依赖项添加到单独的 poetry 依赖项组中。

```
poetry add --group typing my-typing-dep
poetry add --group test my-test-dep
```

最后，让 poetry 使用您的依赖项和您的集成包设置虚拟环境：

```bash
poetry install --with lint,typing,test,test_integration
```

现在您有了一个包含 LangChain 组件模板的新 Python 包！此模板包含每个集成类型的文件，您可以随意复制或删除任何这些文件（包括相关的测试文件）。

要从 [模板] 创建任何单个文件，您可以运行，例如：

```bash
langchain-cli integration new \
    --name parrot-link \
    --name-class ParrotLink \
    --src integration_template/chat_models.py \
    --dst langchain_parrot_link/chat_models_2.py
```

</details>

<details>
    <summary>选项 2：Poetry（手动）</summary>

在本指南中，我们将使用 [Poetry](https://python-poetry.org/) 进行依赖管理和打包，如果您愿意，也可以使用任何其他首选工具。

### **先决条件**

- [GitHub](https://github.com) 账户
- [PyPi](https://pypi.org/) 账户

### 使用 Poetry 引导新的 Python 包

首先，安装 Poetry：

```bash
pip install poetry
```

接下来，为您的包想一个名字。在本指南中，我们将使用 `langchain-parrot-link`。您可以通过在 [PyPi 网站](https://pypi.org/)上搜索来确认该名称在 PyPi 上可用。

接下来，使用 Poetry 创建您的新 Python 包，并使用 `cd` 进入新目录：

```bash
poetry new langchain-parrot-link
cd langchain-parrot-link
```

使用 Poetry 添加主要依赖项，这将把它们添加到您的 `pyproject.toml` 文件中：

```bash
poetry add langchain-core
```

我们还将一些 `test` 依赖项添加到单独的 poetry 依赖项组中。如果您不使用 Poetry，我们建议以一种不会将它们打包到已发布包中的方式添加它们，或者在运行测试时单独安装它们。

`langchain-tests` 将提供我们稍后将使用的 [标准测试](../standard_tests)。我们建议将它们固定到最新版本：<img src="https://img.shields.io/pypi/v/langchain-tests" style={{position:"relative",top:4,left:3}} />

注意：在下方将 `<latest_version>` 替换为 `langchain-tests` 的最新版本。

```bash
poetry add --group test pytest pytest-socket pytest-asyncio langchain-tests==<latest_version>
```

最后，让 poetry 使用您的依赖项和您的集成包设置虚拟环境：

```bash
poetry install --with test
```

现在您已准备好开始编写您的集成包！

### 编写您的集成

假设您正在构建一个简单的集成包，该包为 LangChain 提供了一个简单的 `ChatParrotLink` 聊天模型集成。以下是您的项目结构可能的样子：

```plaintext
langchain-parrot-link/
├── langchain_parrot_link/
│   ├── __init__.py
│   └── chat_models.py
├── tests/
│   ├── __init__.py
│   └── test_chat_models.py
├── pyproject.toml
└── README.md
```

除了 `chat_models.py` 和 `test_chat_models.py` 之外，所有这些文件都应该已经存在！我们将在后面的 [标准测试](../standard_tests) 指南中实现 `test_chat_models.py`。

对于 `chat_models.py`，只需粘贴聊天模型实现的[内容](#implementing-langchain-components)。

</details>

### 将您的包推送到公共 Github 存储库

仅当您想在 LangChain 文档中发布您的集成时才需要此操作。

1. 在 GitHub 上创建一个新的存储库。
2. 将您的代码推送到存储库。
3. 确认您的存储库可供公众查看（例如，在私有浏览窗口中，您未登录 Github）。

您的向量存储实现将取决于您选择的数据库技术。`langchain-core` 包含一个最小的
[内存向量存储](https://python.langchain.com/api_reference/core/vectorstores/langchain_core.vectorstores.in_memory.InMemoryVectorStore.html)，
我们可以将其作为指南。您可以在[此处](https://github.com/langchain-ai/langchain/blob/master/libs/core/langchain_core/vectorstores/in_memory.py)访问代码。

所有向量存储都必须继承自 [VectorStore](https://python.langchain.com/api_reference/core/vectorstores/langchain_core.vectorstores.base.VectorStore.html)
基类。此接口包含用于在向量存储中写入、删除和搜索
文档的方法。

`VectorStore` 支持多种同步和异步搜索类型（例如，
最近邻搜索或最大边际相关性搜索），以及将文档添加到存储的接口。请参阅[API 参考](https://python.langchain.com/api_reference/core/vectorstores/langchain_core.vectorstores.base.VectorStore.html)
了解所有支持的方法。必需的方法如下表所示：

| 方法/属性         | 描述                                         |
|------------------------ |------------------------------------------------------|
| `add_documents`         | 将文档添加到向量存储。                   |
| `delete`                | 从向量存储中删除选定的文档（按 ID） |
| `get_by_ids`            | 从向量存储中获取选定的文档（按 ID）    |
| `similarity_search`     | 获取与查询最相似的文档。               |
| `embeddings` (属性) | 向量存储的嵌入对象。                  |
| `from_texts`            | 通过添加文本实例化向量存储。           |

请注意，`InMemoryVectorStore` 实现了一些可选的搜索类型，以及
用于将对象加载和转储到文件的便捷方法，但这并非所有实现都必需。

[内存向量存储](https://github.com/langchain-ai/langchain/blob/master/libs/core/langchain_core/vectorstores/in_memory.py)

    <TabItem value="tools" label="工具">

工具主要用于两种方式：

1. 定义一个“输入模式”或“参数模式”，用于将工具调用功能传递给聊天模型，并附带文本请求，以便聊天模型可以生成“工具调用”或用于调用工具的参数。
2. 接收如上生成的“工具调用”，并采取行动，返回一个可以作为 ToolMessage 传递回聊天模型的响应。

`Tools` 类必须继承自 [BaseTool](https://python.langchain.com/api_reference/core/tools/langchain_core.tools.base.BaseTool.html#langchain_core.tools.base.BaseTool) 基类。此接口有 3 个属性和 2 个方法应在子类中实现。

| 方法/属性         | 描述                                         |
|------------------------ |------------------------------------------------------|
| `name`                  | 工具的名称（也传递给 LLM）。            |
| `description`           | 工具的描述（也传递给 LLM）。     |
| `args_schema`           | 定义工具输入参数的模式。    |
| `_run`                  | 使用给定参数运行工具。               |
| `_arun`                 | 异步运行工具，使用给定参数。|

### 属性

`name`、`description` 和 `args_schema` 都是应在子类中实现的属性。`name` 和 `description` 是用于标识工具并提供工具功能的描述的字符串。这两者都传递给 LLM，用户可以根据他们使用的 LLM 覆盖这些值，作为“提示工程”的一种形式。为它们提供简洁且 LLM 可用的名称和描述对于工具的初始用户体验很重要。

`args_schema` 是一个 Pydantic `BaseModel`，用于定义工具输入参数的模式。它用于验证传递给工具的输入参数，并为 LLM 在调用工具时填充提供模式。与整个 Tool 类的 `name` 和 `description` 类似，字段的名称（变量名）和描述（`Field(..., description="description")` 的一部分）会传递给 LLM，这些字段中的值应简洁且 LLM 可用。

### 运行方法

`_run` 是应在子类中实现的主要方法。此方法接收来自 `args_schema` 的参数并运行工具，返回字符串响应。此方法通常在 LangGraph 的 [`ToolNode`](https://langchain-ai.github.io/langgraph/how-tos/tool-calling/) 中调用，也可以在旧版 `langchain.agents.AgentExecutor` 中调用。

`_arun` 是可选的，因为默认情况下，`_run` 将在异步执行器中运行。但是，如果您的工具调用任何 API 或执行任何异步工作，您应该实现此方法以除了 `_run` 之外异步运行工具。

    --name parrot-link \
    --name-class ParrotLink \

    <TabItem value="retrievers" label="检索器">

检索器用于根据查询从 API、数据库或其他来源检索文档。`Retriever` 类必须继承自 [BaseRetriever](https://python.langchain.com/api_reference/core/retrievers/langchain_core.retrievers.BaseRetriever.html) 基类。此接口有 1 个属性和 2 个方法应在子类中实现。

| 方法/属性         | 描述                                         |
|------------------------ |------------------------------------------------------|
| `k`                     | 要检索的默认文档数量（可配置）。|
| `_get_relevant_documents`| 根据查询检索文档。                 |
| `_aget_relevant_documents`| 异步根据查询检索文档。  |

### 属性

`k` 是应在子类中实现的属性。此属性可以简单地在类顶部定义，并带有默认值，如 `k: int = 5`。此属性是默认要从检索器检索的文档数量，用户可以在构造或调用检索器时覆盖此值。

### 方法

`_get_relevant_documents` 是应在子类中实现的主要方法。

此方法接收查询并返回 `Document` 对象列表，这些对象具有 2 个主要属性：

- `page_content` - 文档的文本内容
- `metadata` - 关于文档的元数据字典

检索器通常由用户直接调用，例如 `MyRetriever(k=4).invoke("query")`，它将在后台自动调用 `_get_relevant_documents`。

`_aget_relevant_documents` 是可选的，因为默认情况下，`_get_relevant_documents` 将在异步执行器中运行。但是，如果您的检索器调用任何 API 或执行任何异步工作，为了性能起见，您应该实现此方法以除了 `_get_relevant_documents` 之外异步运行检索器。

### 实现

您可以从以下模板或 langchain-cli 命令开始：

```bash
langchain-cli integration new \
    --name parrot-link \
    --name-class ParrotLink \
    --src integration_template/retrievers.py \
    --dst langchain_parrot_link/retrievers.py
```
