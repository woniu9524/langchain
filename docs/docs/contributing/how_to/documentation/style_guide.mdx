# 文档风格指南

随着 LangChain 的不断发展，涵盖各种概念和集成的文档需求也在持续增长。
本页面为撰写 LangChain 文档的任何人提供指导，并概述了我们围绕组织和结构的一些理念。

## 理念

LangChain 的文档遵循 [Diataxis 框架](https://diataxis.fr)。
在此框架下，所有文档都属于以下四类之一：[教程](#tutorials)、[操作指南](#how-to-guides)、[参考资料](#references) 和 [概念解释](#conceptual-guide)。

### 教程

教程是指导读者进行实际操作的课程。其目的是通过一种动手的方式展示实现特定目标的方法，帮助用户理解概念及其交互方式。它们应该**避免**深入探讨实现该目标的多种方法。相反，它应该引导新用户通过推荐的路径来完成教程的目标。虽然教程的最终结果不一定需要完全可用于生产环境，但它应该有用并且能够实际满足教程介绍中明确说明的目标。关于如何处理其他场景的信息应放在操作指南中。

引用 Diataxis 网站的话：

> 教程服务于用户对技能和知识的*习得*——他们的学习。其目的不是帮助用户完成某项工作，而是帮助他们学习。

在 LangChain 中，这些通常是展示端到端用例的更高级别的指南。

一些示例包括：

- [使用 LCEL 构建简单的 LLM 应用](/docs/tutorials/llm_chain/)
- [构建检索增强生成 (RAG) 应用](/docs/tutorials/rag/)

一个好的结构经验法则是遵循这个 [Numpy 示例](https://numpy.org/numpy-tutorials/content/tutorial-svd.html) 的结构。

以下是撰写优秀教程的一些高级技巧：

- 专注于指导用户完成某项工作，但要记住最终目标是传授原理，而不是创建一个完美的生产系统。
- 具体，而非抽象，并遵循一条路径。
  - 无需深入探讨替代方法，但可以引用它们，最好链接到适当的操作指南。
- 尽快“得分”——让用户能够运行并看到输出。
  - 之后可以进行迭代和扩展。
  - 尝试在用户可以运行代码并查看进度的给定步骤处频繁进行检查点。
- 关注结果，而非技术解释。
  - 大量交叉链接到相关的概念/参考页面。
- 第一次提及 LangChain 概念时，使用其全称（例如，“LangChain Expression Language (LCEL)”），并链接到其概念/其他文档页面。
  - 添加一个链接到具有必要背景信息的页面的先决条件提示也很有帮助。
- 以总结/后续步骤部分结束，总结教程内容和未来阅读内容，例如相关的操作指南。

### 操作指南

顾名思义，操作指南演示如何执行离散且特定的操作。
它应该假定用户已经熟悉底层概念，并专注于解决当前问题。但是，
它仍应提供一些背景信息或列出信息可能相关的特定场景。
在某些情况下，它们可以并且应该讨论替代方案，因为一种方法可能比另一种更好。

引用 Diataxis 网站的话：

> 操作指南服务于已有能力的用户的*工作*，您可以假设他们知道自己想做什么，并且能够正确地遵循您的指示。

一些示例包括：

- [如何：从模型返回结构化数据](/docs/how_to/structured_output/)
- [如何：编写自定义聊天模型](/docs/how_to/custom_chat_model/)

以下是撰写优秀操作指南的一些高级技巧：

- 在开头清晰地解释您正在指导用户完成什么。
- 假设比教程更高的意图，并展示用户需要做什么才能完成该任务。
- 假定熟悉概念，但解释建议的操作为何有帮助。
  - 大量交叉链接到概念/参考页面。
- 讨论在解决问题时可能出现的替代方案和现实权衡。
- 使用大量示例代码。
  - 优先使用读者可以复制和运行的完整代码块。
- 以总结/后续步骤部分结束，总结教程内容和未来阅读内容，例如其他相关的操作指南。

### 概念指南

LangChain 的概念指南属于 Diataxis 的**解释**象限。这些指南应以比操作指南或教程更抽象的方式涵盖 LangChain 的术语和概念，面向有兴趣更深入理解和洞察框架的好奇用户。尽量避免过多的代码示例，因为主要目标是为用户提供视角，而不是完成一个实际项目。这些指南应涵盖事物为何以这种方式工作。

本关于文档风格的指南旨在归入此类。

引用 Diataxis 网站的话：

> 解释的视角比其他类型更高更广。它不像操作指南那样采用用户的视平线视角，也不像参考资料那样近距离观察机器。它在每种情况下都涵盖一个主题——“一个知识领域”，该领域必须以合理、有意义的方式进行界定。

一些示例包括：

- [检索概念文档](/docs/concepts/retrieval)
- [聊天模型概念文档](/docs/concepts/chat_models)

以下是撰写优秀概念指南的一些高级技巧：

- 解释设计决策。概念 X 为什么存在以及为什么这样设计？
- 使用类比并引用其他概念和替代方案
- 避免过多地混合参考内容
- 您可以并且应该引用其他指南中涵盖的内容，但请确保链接到它们

### 参考资料

参考资料包含详细的、低级的信息，准确描述了存在哪些功能以及如何使用它们。
在 LangChain 中，这主要是我们的 API 参考页面，这些页面是从代码中的文档字符串填充的。
参考页面通常不会从头到尾阅读，而是在用户需要了解如何使用特定内容时根据需要查阅。

引用 Diataxis 网站的话：

> 参考指南的唯一目的是尽可能简洁、有序地进行描述。教程和操作指南的内容由用户的需求驱动，而参考资料的内容则由其描述的产品驱动。

LangChain 中的许多参考页面都是从代码自动生成的，
但以下是撰写良好文档字符串的一些高级技巧：

- 保持简洁
- 讨论特殊情况和与用户期望的偏差
- 详细说明必需的输入和输出
- 简要说明何时可能使用该功能是可以的，但深入的细节应放在其他部分。

每个类别都有其独特的目的，并需要特定的内容编写和组织方法。

## 通用指南

以下是您在编写和组织文档时应考虑的其他一些指南。

我们通常不合并来自外部贡献者的新教程，除非有迫切需求。
我们欢迎更新以及新的集成文档、操作指南和参考资料。

### 避免重复

多个页面深入涵盖相同内容的维护困难且容易引起混淆。对于给定的概念或功能，应该只有一个（极少数情况下为两个）规范页面。相反，您应该链接到其他指南。

### 链接到其他部分

由于文档的各个部分并非孤立存在，因此频繁链接到其他部分非常重要，以便开发人员在阅读流程中了解更多关于不熟悉的主题。

这包括链接到 API 参考和概念部分！

### 保持简洁

总的来说，采取少即是多的方法。如果存在一个包含概念良好解释的其他部分，您应该链接到它，而不是重新解释它，除非您正在记录的概念引入了一些新的细节。

保持简洁，包括在代码示例中。

### 通用风格

- 尽可能使用主动语态和现在时态
- 使用示例和代码片段来说明概念和用法
- 使用适当的标题级别（`#`、`##`、`###` 等）来分层组织内容
- 使用更少的单元格和更多的代码，以便于复制粘贴
- 使用项目符号和编号列表将信息分解为易于理解的块
- 经常使用表格（尤其是在**参考资料**部分）和图表来直观地呈现信息
- 为较长的文档页面包含目录，以帮助读者导航内容，但为较短的页面隐藏它
