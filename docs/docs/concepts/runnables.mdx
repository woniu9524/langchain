# Runnable 接口

Runnable 接口是使用 LangChain 组件的基础，它被许多组件实现，例如 [语言模型](/docs/concepts/chat_models)、[输出解析器](/docs/concepts/output_parsers)、[检索器](/docs/concepts/retrievers)、[编译后的 LangGraph 图](
https://langchain-ai.github.io/langgraph/concepts/low_level/#compiling-your-graph) 等等。

本指南涵盖了 Runnable 接口的主要概念和方法，它允许开发者以一致且可预测的方式与各种 LangChain 组件进行交互。

:::info 相关资源
* ["Runnable" 接口 API 参考](https://python.langchain.com/api_reference/core/runnables/langchain_core.runnables.base.Runnable.html#langchain_core.runnables.base.Runnable) 提供了 Runnable 接口及其方法的详细概述。
* LangChain Core API 参考中可以找到内置 `Runnables` 的列表。许多这些 Runnables 在使用 [LangChain 表达式语言 (LCEL)](/docs/concepts/lcel) 组合自定义“链”时非常有用。
:::

## Runnable 接口概述

Runnable 方式定义了一个标准接口，允许 Runnable 组件：

* [调用](/docs/how_to/lcel_cheatsheet/#invoke-a-runnable)：单个输入被转换为输出。
* [批量处理](/docs/how_to/lcel_cheatsheet/#batch-a-runnable)：多个输入被高效地转换为输出。
* [流式传输](/docs/how_to/lcel_cheatsheet/#stream-a-runnable)：输出在生成时被流式传输。
* 自检：可以访问 Runnable 的输入、输出和配置的模式信息。
* 组合：可以使用 [LangChain 表达式语言 (LCEL)](/docs/concepts/lcel) 将多个 Runnable 组合在一起以创建复杂的管道。

请查阅 [LCEL 备忘单](/docs/how_to/lcel_cheatsheet) 以了解涉及 Runnable 接口和 LCEL 表达式的常见模式。

### 优化的并行执行（批量）
<span data-heading-keywords="batch"></span>

LangChain Runnables 提供内置的 `batch`（和 `batch_as_completed`）API，允许您并行处理多个输入。

使用这些方法可以显著提高处理多个独立输入时的性能，因为处理可以并行而不是顺序进行。

两种批量选项是：

* `batch`：并行处理多个输入，并按输入顺序返回结果。
* `batch_as_completed`：并行处理多个输入，并按完成顺序返回结果。结果可能乱序，但每个结果都包含输入索引以进行匹配。

`batch` 和 `batch_as_completed` 的默认实现使用线程池执行器并行运行 `invoke` 方法。这允许高效的并行执行，而无需用户管理线程，并加速了 I/O 密集型（例如，进行 API 请求、读取文件等）的代码。对于 CPU 密集型操作，它的效果不会那么好，因为 Python 中的 GIL（全局解释器锁）会阻止真正的并行执行。

某些 Runnables 可能会提供自己的 `batch` 和 `batch_as_completed` 实现，以针对其特定用例进行优化（例如，依赖于模型提供商提供的 `batch` API）。

:::note
`abatch` 和 `abatch_as_completed` 的异步版本依赖于 asyncio 的 [gather](https://docs.python.org/3/library/asyncio-task.html#asyncio.gather) 和 [as_completed](https://docs.python.org/3/library/asyncio-task.html#asyncio.as_completed) 函数来并行运行 `ainvoke` 方法。
:::

:::tip
当使用 `batch` 或 `batch_as_completed` 处理大量输入时，用户可能希望控制并行调用的最大数量。这可以通过在 `RunnableConfig` 字典中设置 `max_concurrency` 属性来完成。有关更多信息，请参阅 [RunnableConfig](/docs/concepts/runnables/#runnableconfig)。

聊天模型还内置了 [速率限制器](/docs/concepts/chat_models#rate-limiting)，可用于控制请求的速率。
:::

### 异步支持
<span data-heading-keywords="async-api"></span>

Runnables 公开了异步 API，允许使用 Python 中的 `await` 语法进行调用。异步方法可以通过 "a" 前缀（例如 `ainvoke`、`abatch`、`astream`、`abatch_as_completed`）来识别。

有关更多详细信息，请参阅 [LangChain 异步编程](/docs/concepts/async) 指南。

## 流式传输 API
<span data-heading-keywords="streaming-api"></span>

流式传输对于使基于 LLM 的应用程序对最终用户感觉响应至关重要。

Runnables 公开了以下三个流式传输 API：

1. 同步 [stream](https://python.langchain.com/api_reference/core/runnables/langchain_core.runnables.base.Runnable.html#langchain_core.runnables.base.Runnable.stream) 和异步 [astream](https://python.langchain.com/api_reference/core/runnables/langchain_core.runnables.base.Runnable.html#langchain_core.runnables.base.Runnable.astream)：在生成时产生 Runnable 的输出。
2. 异步 `astream_events`：一个更高级的流式传输 API，允许流式传输中间步骤和最终输出。
3. **遗留**异步 `astream_log`：一个遗留的流式传输 API，用于流式传输中间步骤和最终输出。

有关如何在 LangChain 中进行流式传输的更多详细信息，请参阅 [流式传输概念指南](/docs/concepts/streaming)。

## 输入和输出类型

每个 `Runnable` 都由输入和输出类型来表征。这些输入和输出类型可以是任何 Python 对象，并由 Runnable 本身定义。

导致 Runnable 执行的方法（例如 `invoke`、`batch`、`stream`、`astream_events`）使用这些输入和输出类型。

* `invoke`：接受一个输入并返回一个输出。
* `batch`：接受一个输入列表并返回一个输出列表。
* `stream`：接受一个输入并返回一个生成器，该生成器产生输出。

**输入类型**和**输出类型**因组件而异：

| 组件    | 输入类型                                       | 输出类型           |
|--------------|--------------------------------------------------|-----------------------|
| Prompt       | 字典                                             | PromptValue           |
| ChatModel    | 字符串、聊天消息列表或 PromptValue               | ChatMessage           |
| LLM          | 字符串、聊天消息列表或 PromptValue               | String                |
| OutputParser | LLM 或 ChatModel 的输出                          | 取决于解析器        |
| Retriever    | 字符串                                           | Document 列表         |
| Tool         | 字符串或字典，取决于工具                         | 取决于工具            |

有关输入和输出类型及其使用方法的更多信息，请参阅各个组件的文档。

### 自检模式

:::note
这是一个高级功能，大多数用户都不需要。您可能应该跳过此部分，除非您有特定的检查 Runnable 模式的需求。
:::

在更高级的用例中，您可能希望以编程方式**检查** Runnable，并确定 Runnable 期望和生成的输入和输出类型。

Runnable 接口提供了获取 Runnable 输入和输出类型的 [JSON Schema](https://json-schema.org/) 以及输入和输出类型的 [Pydantic Schema](https://docs.pydantic.dev/latest/) 的方法。

这些 API 主要在内部用于单元测试，以及由 [LangServe](/docs/concepts/architecture#langserve) 使用这些 API 进行输入验证和生成 [OpenAPI 文档](https://www.openapis.org/)。

除了输入和输出类型之外，一些 Runnables 还设置了额外的运行时配置选项。
有相应的 API 可以获取 Runnable 的配置选项的 Pydantic Schema 和 JSON Schema。
有关更多信息，请参阅 [可配置 Runnable](#configurable-runnables) 部分。

| 方法                  | 描述                                                      |
|-------------------------|-----------------------------------------------------------|
| `get_input_schema`      | 提供 Runnable 输入模式的 Pydantic Schema。                |
| `get_output_schema`      | 提供 Runnable 输出模式的 Pydantic Schema。                |
| `config_schema`         | 提供 Runnable 配置模式的 Pydantic Schema。                |
| `get_input_jsonschema`  | 提供 Runnable 输入模式的 JSONSchema。                     |
| `get_output_jsonschema` | 提供 Runnable 输出模式的 JSONSchema。                     |
| `get_config_jsonschema` | 提供 Runnable 配置模式的 JSONSchema。                     |


#### With_types

LangChain 将自动尝试根据可用信息推断 Runnable 的输入和输出类型。

目前，对于使用 [LCEL](/docs/concepts/lcel) 组合构建的更复杂的 Runnable，此推断效果不佳，推断出的输入和/或输出类型可能不正确。在这种情况下，我们建议用户使用 `with_types` 方法（[API 参考](https://python.langchain.com/api_reference/core/runnables/langchain_core.runnables.base.Runnable.html#langchain_core.runnables.base.Runnable.with_types)）覆盖推断出的输入和输出类型。

## RunnableConfig

用于执行 Runnable 的任何方法（例如 `invoke`、`batch`、`stream`、`astream_events`）都接受第二个参数，称为 `RunnableConfig`（[API 参考](https://python.langchain.com/api_reference/core/runnables/langchain_core.runnables.config.RunnableConfig.html#RunnableConfig)）。此参数是一个字典，其中包含将在 Runnable 执行期间在运行时使用的 Runnable 的配置。

`RunnableConfig` 可以定义以下任何属性：

| 属性       | 描述                                                                                |
|-----------------|--------------------------------------------------------------------------------------------|
| `run_name`        | 为给定 Runnable 使用的名称（不继承）。                                          |
| `run_id`          | 此调用的唯一标识符。子调用将获得自己的唯一 run id。              |
| `tags`            | 此调用及任何子调用的标签。                                                      |
| `metadata`        | 此调用及任何子调用的元数据。                                                  |
| `callbacks`       | 此调用及任何子调用的回调。                                                  |
| `max_concurrency` | 要进行的并行调用的最大数量（例如，由 batch 使用）。                            |
| `recursion_limit` | 调用可以递归的最大次数（例如，由返回 Runnables 的 Runnables 使用） |
| `configurable`    | Runnable 的可配置属性的运行时值。                               |

将 `config` 传递给 `invoke` 方法如下所示：

```python
some_runnable.invoke(
   some_input,
   config={
      'run_name': 'my_run',
      'tags': ['tag1', 'tag2'],
      'metadata': {'key': 'value'}

   }
)
```

### RunnableConfig 的传播

许多 `Runnables` 由其他 Runnables 组成，并且 `RunnableConfig` 必须传播到 Runnable 所做的所有子调用中，这一点很重要。这允许为父 Runnable 提供运行时配置值，这些值将被所有子调用继承。

如果不是这样，将无法设置和传播在 `RunnableConfig` 中定义的 [回调](/docs/concepts/callbacks) 或其他配置值，如 `tags` 和 `metadata`，这些值是所有子调用都期望继承的。

创建新 `Runnables` 的两种主要模式是：

1. 使用 [LangChain 表达式语言 (LCEL)](/docs/concepts/lcel) 声明式创建：

    ```python
    chain = prompt | chat_model | output_parser
    ```

2. 使用 [自定义 Runnable](#custom-runnables)（例如 `RunnableLambda`）或使用 `@tool` 装饰器：

    ```python
    def foo(input):
        # 注意这里直接使用了 .invoke()
        return bar_runnable.invoke(input)
    foo_runnable = RunnableLambda(foo)
    ```

LangChain 将尝试自动为这两种模式传播 `RunnableConfig`。

对于处理第二种模式，LangChain 依赖于 Python 的 [contextvars](https://docs.python.org/3/library/contextvars.html)。

在 Python 3.11 及更高版本中，这开箱即用，您无需执行任何特殊操作即可将 `RunnableConfig` 传播到子调用。

在 Python 3.9 和 3.10 中，如果您使用的是**异步代码**，则需要手动将 `RunnableConfig` 传递给调用时的 `Runnable`。

这是由于 Python 3.9 和 3.10 中 [asyncio 任务](https://docs.python.org/3/library/asyncio-task.html#asyncio.create_task) 的限制，它不接受 `context` 参数。

手动传播 `RunnableConfig` 如下所示：

```python
async def foo(input, config): # <-- 注意 config 参数
    return await bar_runnable.ainvoke(input, config=config)

foo_runnable = RunnableLambda(foo)
```

:::caution
当使用 Python 3.10 或更低版本编写异步代码时，`RunnableConfig` 无法自动传播，您需要手动进行！这是使用 `astream_events` 和 `astream_log` 流式传输数据时常见的陷阱，因为这些方法依赖于正确传播 `RunnableConfig` 中定义的 [回调](/docs/concepts/callbacks)。
:::

### 设置自定义运行名称、标签和元数据

`RunnableConfig` 字典的 `run_name`、`tags` 和 `metadata` 属性可用于为给定运行设置自定义运行名称、标签和元数据。

`run_name` 是一个字符串，可用于为运行设置自定义名称。此名称将在日志和其他位置用于标识运行。它不会被子调用继承。

`tags` 和 `metadata` 属性分别是列表和字典，可用于为运行设置自定义标签和元数据。这些值将被子调用继承。

使用这些属性对于跟踪和调试运行非常有用，因为它们将在 [LangSmith](https://docs.smith.langchain.com/) 中显示为可供筛选和搜索的跟踪属性。

这些属性还将传播到 [回调](/docs/concepts/callbacks)，并作为流中每个事件的一部分出现在 [astream_events](/docs/concepts/streaming) 等流式传输 API 中。

:::note 相关
* [如何使用 LangChain 进行跟踪](https://docs.smith.langchain.com/how_to_guides/tracing/trace_with_langchain)
:::

### 设置运行 ID

:::note
这是一个高级功能，大多数用户都不需要。
:::

您可能需要为给定的运行设置自定义 `run_id`，以防您想稍后引用它或将其与其他系统关联。

`run_id` 必须是有效的 UUID 字符串，并且对于每次运行都是**唯一**的。它用于标识父运行，子调用将自动获得自己的唯一 run id。

要设置自定义 `run_id`，您可以在调用 Runnable 时将其作为键值对传递到 `config` 字典中：

```python
import uuid

run_id = uuid.uuid4()

some_runnable.invoke(
   some_input,
   config={
      'run_id': run_id
   }
)

# 使用 run_id 做一些事情
```

### 设置递归限制

:::note
这是一个高级功能，大多数用户都不需要。
:::

某些 Runnables 可能会返回其他 Runnables，如果处理不当，可能会导致无限递归。为防止这种情况，您可以在 `RunnableConfig` 字典中设置 `recursion_limit`。这将限制 Runnable 可以递归的次数。

### 设置最大并发数

如果使用 `batch` 或 `batch_as_completed` 方法，您可以在 `RunnableConfig` 字典中设置 `max_concurrency` 属性来控制要进行的并行调用的最大数量。当您想限制并行调用的数量以防止服务器或 API 过载时，这会很有用。


:::tip
如果您试图限制**聊天模型**发起的请求数量，您可以使用内置的 [速率限制器](/docs/concepts/chat_models#rate-limiting) 而不是设置 `max_concurrency`，这样会更有效。

有关更多信息，请参阅 [如何处理速率限制](/docs/how_to/chat_model_rate_limiting/) 指南。
:::

### 设置可配置项

`configurable` 字段用于为 Runnable 的可配置属性传递运行时值。

它经常在 [LangGraph](/docs/concepts/architecture#langgraph) 中与 [LangGraph 持久化](https://langchain-ai.github.io/langgraph/concepts/persistence/) 和 [内存](https://langchain-ai.github.io/langgraph/concepts/memory/) 一起使用。

它用于类似的目的，在 [RunnableWithMessageHistory](https://python.langchain.com/api_reference/core/runnables/langchain_core.runnables.history.RunnableWithMessageHistory.html#langchain_core.runnables.history.RunnableWithMessageHistory) 中指定 `session_id` / `conversation_id` 以跟踪对话历史记录。

此外，您还可以使用它来指定任何自定义配置选项，以传递给它们创建的任何 [可配置 Runnable](#configurable-runnables)。

### 设置回调

使用此选项可在运行时为 Runnable 配置 [回调](/docs/concepts/callbacks)。这些回调将传递给 Runnable 进行的所有子调用。

```python
some_runnable.invoke(
   some_input,
   {
      "callbacks": [
         SomeCallbackHandler(),
         AnotherCallbackHandler(),
      ]
   }
)
```

有关如何在 LangChain 中使用回调的更多信息，请阅读 [回调概念指南](/docs/concepts/callbacks)。

:::important
如果您在异步环境中使用 Python 3.9 或 3.10，则在某些情况下必须手动将 `RunnableConfig` 传播到子调用。有关更多信息，请参阅 [传播 RunnableConfig](#propagation-of-runnableconfig) 部分。
:::

## 从函数创建 Runnable {#custom-runnables}

您可能需要创建一个运行任意逻辑的自定义 Runnable。如果您使用 [LangChain 表达式语言 (LCEL)](/docs/concepts/lcel) 来组合多个 Runnables，并且需要在其中一个步骤中添加自定义处理逻辑，那么这尤其有用。

有两种方法可以从函数创建自定义 Runnable：

* `RunnableLambda`：适用于不需要流式传输的简单转换。
* `RunnableGenerator`：适用于需要流式传输的更复杂的转换。

有关如何使用 `RunnableLambda` 和 `RunnableGenerator` 的更多信息，请参阅 [如何运行自定义函数](/docs/how_to/functions) 指南。

:::important
用户不应尝试通过子类化 Runnables 来创建新的自定义 Runnable。这比仅使用 `RunnableLambda` 或 `RunnableGenerator` 要复杂得多，也更容易出错。
:::

## 可配置的 Runnable

:::note
这是一个高级功能，大多数用户都不需要。

它有助于配置使用 [LangChain 表达式语言 (LCEL)](/docs/concepts/lcel) 创建的大型“链”，并由 [LangServe](/docs/concepts/architecture#langserve) 用于已部署的 Runnables。
:::

有时您可能希望试验，甚至向最终用户公开多种做事方式。这可能涉及调整聊天模型的温度等参数，甚至在不同的聊天模型之间切换。

为简化此过程，Runnable 接口提供了两种在运行时创建可配置 Runnable 的方法：

* `configurable_fields`：此方法允许您配置 Runnable 中的特定**属性**。例如，聊天模型的 `temperature` 属性。
* `configurable_alternatives`：此方法使您能够指定在运行时可以运行的**替代** Runnable。例如，您可以指定一组不同的聊天模型。

有关如何配置运行时链内部的更多信息，请参阅 [如何配置运行时链内部](/docs/how_to/configure) 指南。
