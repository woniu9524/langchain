# 流式传输

:::info 先决条件
* [可运行接口](/docs/concepts/runnables)
* [聊天模型](/docs/concepts/chat_models)
:::

**流式传输**对于提高基于[LLM](/docs/concepts/chat_models)构建的应用程序的响应能力至关重要。通过逐步显示输出，甚至在完整响应准备好之前，流式传输即可显著改善用户体验 (UX)，尤其是在处理 LLM 的延迟时。

## 概述

从[LLM](/docs/concepts/chat_models)生成完整响应通常需要几秒钟的延迟，在具有多个模型调用的复杂应用程序中，这种延迟会更加明显。幸运的是，LLM 会迭代地生成响应，允许在生成中间结果时显示它们。通过流式传输这些中间输出，LangChain 可在 LLM 驱动的应用中实现更流畅的用户体验，并在其设计的核心提供对流式传输的内置支持。

在本指南中，我们将讨论 LLM 应用中的流式传输，并探讨 LangChain 的流式传输 API 如何促进应用程序中各种组件的实时输出。

## 在 LLM 应用中流式传输什么

在涉及 LLM 的应用程序中，可以流式传输多种类型的数据，以通过减少感知延迟和提高透明度来改善用户体验。这些包括：

### 1. 流式传输 LLM 输出

流式传输最常见且最关键的数据是 LLM 本身生成的输出。LLM 通常需要一段时间才能生成完整响应，通过实时流式传输输出，用户可以看到正在生成的中间结果。这提供了即时反馈，并有助于减少用户的等待时间。

### 2. 流式传输管道或工作流进度

除了流式传输 LLM 输出之外，流式传输更复杂的工作流或管道的进度也很有用，让用户对应用程序的整体进度有所了解。这可能包括：

- **在 LangGraph 工作流中：**
使用 [LangGraph](/docs/concepts/architecture#langgraph)，工作流由代表各种步骤的节点和边组成。这里的流式传输涉及跟踪**图状态**的变化，因为各个**节点**请求更新。这允许更精细地监控工作流中当前活动的节点，从而在工作流通过不同阶段时提供有关工作流状态的实时更新。

- **在 LCEL 管道中：**
流式传输 [LCEL](/docs/concepts/lcel) 管道的更新涉及捕获来自各个**子可运行组件**的进度。例如，当管道的不同步骤或组件执行时，您可以流式传输当前正在运行的子可运行组件，从而提供对整个管道进度的实时洞察。

流式传输管道或工作流进度对于为用户提供应用程序在执行过程中的位置的清晰视图至关重要。

### 3. 流式传输自定义数据

在某些情况下，您可能需要流式传输**自定义数据**，这些数据超出了管道或工作流结构提供的信息。这些自定义信息被注入到工作流中的特定步骤，无论该步骤是工具还是 LangGraph 节点。例如，您可以实时流式传输有关工具正在做什么的更新，或者 LangGraph 节点的进度。这些直接从步骤内部发出的精细数据提供了对工作流执行的更详细的见解，并且在需要更多可见性的复杂过程中特别有用。

## 流式传输 API

LangChain 提供了两个主要的 API，用于实时流式传输输出。这些 API 支持实现[可运行接口](/docs/concepts/runnables)的任何组件，包括[LLM](/docs/concepts/chat_models)、[编译的 LangGraph 图](https://langchain-ai.github.io/langgraph/concepts/low_level/)以及使用[LCEL](/docs/concepts/lcel)生成的任何可运行组件。

1. 同步 [stream](https://python.langchain.com/api_reference/core/runnables/langchain_core.runnables.base.Runnable.html#langchain_core.runnables.base.Runnable.stream) 和异步 [astream](https://python.langchain.com/api_reference/core/runnables/langchain_core.runnables.base.Runnable.html#langchain_core.runnables.base.Runnable.astream)：用于在生成时流式传输单个可运行组件（例如，聊天模型）的输出，或流式传输使用 LangGraph 创建的任何工作流。
2. 仅异步的 [astream_events](https://python.langchain.com/api_reference/core/runnables/langchain_core.runnables.base.Runnable.html#langchain_core.runnables.base.Runnable.astream_events)：使用此 API 可访问完全使用[LCEL](/docs/concepts/lcel)构建的 LLM 应用中的自定义事件和中间输出。请注意，此 API 可用，但在使用 LangGraph 时不需要。

:::note
此外，还有一个**旧版**异步 [astream_log](https://python.langchain.com/api_reference/core/runnables/langchain_core.runnables.base.Runnable.html#langchain_core.runnables.base.Runnable.astream_log) API。不建议在新项目中使用此 API，因为它比其他流式传输 API 更复杂且功能更少。
:::

### `stream()` 和 `astream()`

`stream()` 方法返回一个迭代器，该迭代器在生成输出时同步产生输出块。您可以使用 `for` 循环实时处理每个块。例如，在使用 LLM 时，这允许在生成输出时以增量方式流式传输输出，从而减少用户的等待时间。

`stream()` 和 `astream()` 方法生成的块的类型取决于正在流式传输的组件。例如，当从 [LLM](/docs/concepts/chat_models) 流式传输时，每个组件将是 [AIMessageChunk](/docs/concepts/messages#aimessagechunk)；但是，对于其他组件，块可能不同。

`stream()` 方法返回一个迭代器，该迭代器在生成块时产生这些块。例如，

```python
for chunk in component.stream(some_input):
    # 重要提示：尽量保持对每个块的处理尽可能高效。
    # 当您处理当前块时，上游组件正在等待生成下一个块。
    # 例如，在使用 LangGraph 时，图执行会暂停，直到当前块被处理。
    # 在极端情况下，这甚至可能导致超时（例如，当 LLM 输出
    # 从具有超时的 API 流式传输时）。
    print(chunk)
```

[异步版本](/docs/concepts/async) `astream()` 的工作方式类似，但专为非阻塞工作流而设计。您可以在异步代码中使用它来实现相同的实时流式传输行为。

#### 与聊天模型一起使用

当与聊天模型一起使用 `stream()` 或 `astream()` 时，输出将作为 [AIMessageChunks](/docs/concepts/messages#aimessagechunk) 在生成时进行流式传输。这允许您在 LLM 生成输出时以增量方式显示或处理 LLM 的输出，这在交互式应用程序或界面中尤其有用。

#### 与 LangGraph 一起使用

[LangGraph](/docs/concepts/architecture#langgraph) 编译的图是[可运行组件](/docs/concepts/runnables)，支持标准的流式传输 API。

当与 LangGraph 一起使用 *stream* 和 *astream* 方法时，您可以选择**一个或多个**[流式传输模式](https://langchain-ai.github.io/langgraph/reference/types/#langgraph.types.StreamMode)，这些模式允许您控制要流式传输的输出类型。可用的流式传输模式包括：

- **"values"**：为每个步骤发出[状态](https://langchain-ai.github.io/langgraph/concepts/low_level/)的所有值。
- **"updates"**：仅发出节点名称和节点在每个步骤后返回的更新。
- **"debug"**：为每个步骤发出调试事件。
- **"messages"**：逐个[令牌](https://langchain-ai.github.io/langgraph/concepts/tokens)地发出 LLM [消息](/docs/concepts/messages)。
- **"custom"**：发出使用[LangGraph 的 StreamWriter](https://langchain-ai.github.io/langgraph/reference/types/#langgraph.types.StreamWriter) 编写的自定义输出。

有关更多信息，请参阅：
* [LangGraph 流式传输概念指南](https://langchain-ai.github.io/langgraph/concepts/streaming/)，了解有关与 LangGraph 一起流式传输的更多信息。
* [LangGraph 流式传输操作指南](https://langchain-ai.github.io/langgraph/how-tos/#streaming)，了解 LangGraph 中流式传输的具体示例。

#### 与 LCEL 一起使用

如果您使用[LangChain 的表达式语言 (LCEL)](/docs/concepts/lcel) 组合多个可运行组件，则 `stream()` 和 `astream()` 方法将按照约定流式传输链中最后一个组件的输出。这允许以增量方式流式传输最终处理的结果。**LCEL** 会尝试优化管道中的流式传输延迟，以便尽快获得最后一个组件的流式传输结果。



### `astream_events`
<span data-heading-keywords="astream_events,stream_events,stream events"></span>

:::tip
使用 `astream_events` API 来访问完全使用[LCEL](/docs/concepts/lcel)构建的 LLM 应用中的自定义数据和中间输出。

虽然此 API 也可用于[LangGraph](/docs/concepts/architecture#langgraph)，但在使用 LangGraph 时通常不需要它，因为 `stream` 和 `astream` 方法为 LangGraph 图提供了全面的流式传输功能。
:::

对于使用**LCEL**构建的链，`.stream()` 方法仅流式传输链中最后一个组件的输出。这对于某些应用程序来说可能足够了，但随着您构建更复杂的链（将多个 LLM 调用组合在一起），您可能希望使用链的中间值以及最终输出。例如，在构建文档问答应用时，您可能希望在最终生成的同时返回源。

有几种方法可以做到这一点，例如[使用回调](/docs/concepts/callbacks)，或者通过以某种方式构建链，使其能够将中间值传递到末尾，例如通过链式[`.assign()`](/docs/how_to/passthrough/)调用，但 LangChain 还包含一个 `.astream_events()` 方法，该方法将回调的灵活性与 `.stream()` 的易用性相结合。调用时，它会返回一个迭代器，该迭代器会产生您可以根据项目需求进行过滤和处理的[各种类型的事件](/docs/how_to/streaming/#event-reference)。

这是一个只打印包含流式聊天模型输出的事件的小示例：

```python
from langchain_core.output_parsers import StrOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_anthropic import ChatAnthropic

model = ChatAnthropic(model="claude-3-sonnet-20240229")

prompt = ChatPromptTemplate.from_template("tell me a joke about {topic}")
parser = StrOutputParser()
chain = prompt | model | parser

async for event in chain.astream_events({"topic": "parrot"}):
    kind = event["event"]
    if kind == "on_chat_model_stream":
        print(event, end="|", flush=True)
```

您可以大致将其视为回调事件的迭代器（尽管格式不同）——并且您可以将其用于几乎所有 LangChain 组件！

有关如何使用 `.astream_events()` 的更详细信息，包括列出可用事件的表格，请参阅[本指南](/docs/how_to/streaming/#using-stream-events)。

## 将自定义数据写入流

要将自定义数据写入流，您需要根据正在使用的组件选择以下方法之一：

1. LangGraph 的[StreamWriter](https://langchain-ai.github.io/langgraph/reference/types/#langgraph.types.StreamWriter) 可用于写入自定义数据，这些数据将在使用 LangGraph 时通过 **stream** 和 **astream** API 显示。**重要提示**：这是 LangGraph 的一项功能，因此在处理纯 LCEL 时不可用。有关更多信息，请参阅[如何流式传输自定义数据](https://langchain-ai.github.io/langgraph/how-tos/streaming-content/)。
2. [dispatch_events](https://python.langchain.com/api_reference/core/callbacks/langchain_core.callbacks.manager.dispatch_custom_event.html#) / [adispatch_events](https://python.langchain.com/api_reference/core/callbacks/langchain_core.callbacks.manager.adispatch_custom_event.html) 可用于写入将通过 **astream_events** API 显示的自定义数据。有关更多信息，请参阅[如何分派自定义回调事件](/docs/how_to/callbacks_custom_events/#astream-events-api)。

## “自动流式传输”聊天模型

LangChain 通过在某些情况下自动启用流式传输模式来简化聊天模型的流式传输，即使您没有显式调用流式传输方法。当您使用非流式 `invoke` 方法但仍希望流式传输整个应用程序（包括聊天模型的中间结果）时，这尤其有用。

### 工作原理

当您在聊天模型上调用 `invoke`（或 `ainvoke`）方法时，如果 LangChain 检测到您正在尝试流式传输整个应用程序，它将自动切换到流式传输模式。

在底层，它将使用 `invoke`（或 `ainvoke`）来使用 `stream`（或 `astream`）方法来生成其输出。从使用 `invoke` 的代码来看，调用的结果是相同的；但是，在流式传输聊天模型时，LangChain 会负责调用 LangChain 的[回调系统](/docs/concepts/callbacks)中的 `on_llm_new_token` 事件。这些回调事件允许 LangGraph 的 `stream`/`astream` 和 `astream_events` 实时显示聊天模型的输出。

示例：

```python
def node(state):
    ...
    # 以下代码使用 invoke 方法，但 LangChain 将
    # 自动切换到流式传输模式
    # 当它检测到整个
    # 应用程序正在被流式传输时。
    ai_message = model.invoke(state["messages"])
    ...

for chunk in compiled_graph.stream(..., mode="messages"):
    ...
```
## 异步编程

LangChain 提供了许多方法的同步 (sync) 和异步 (async) 版本。异步方法通常以“a”作为前缀（例如 `ainvoke`、`astream`）。在编写异步代码时，必须一致地使用这些异步方法，以确保非阻塞行为和最佳性能。

如果流式传输的数据未能实时出现，请确保您为工作流使用了正确的异步方法。

请参阅[LangChain 中的异步编程指南](/docs/concepts/async)，了解有关使用 LangChain 编写异步代码的更多信息。
