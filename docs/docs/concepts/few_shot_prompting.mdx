# 少样本提示 (Few-shot prompting)

:::note 先决条件

- [对话模型](/docs/concepts/chat_models/)
:::

## 概述

提高模型性能最有效的方法之一是向模型提供它应该做什么的示例。将示例输入和预期输添加到模型提示中的技术被称为“少样本提示”。该技术基于 [Language Models are Few-Shot Learners](https://arxiv.org/abs/2005.14165) 这篇论文。在进行少样本提示时，有几点需要考虑：

1. 示例是如何生成的？
2. 每个提示中有多少示例？
3. 示例如何在运行时选择？
4. 示例在提示中是如何格式化的？

以下是每个方面的考虑因素。

## 1. 生成示例

少样本提示的第一步也是最重要的一步是创建一个良好的示例数据集。好的示例在运行时应该是相关的、清晰的、信息丰富的，并提供模型尚未了解的信息。

从高层次来看，生成示例的基本方法有：
- 手动：一个人或多人生成他们认为有用的示例。
- 更好的模型：使用一个更好的（通常更昂贵/更慢）模型的响应作为示例，供一个较差的（通常更便宜/更快）模型使用。
- 用户反馈：用户（或标注者）对与应用程序的交互留下反馈，并根据该反馈生成示例（例如，所有带有积极反馈的交互都可以转化为示例）。
- LLM 反馈：与用户反馈相同，但通过让模型自行评估来自动化该过程。

哪种方法最好取决于您的任务。对于需要非常深入地理解少量核心原则的任务，精心制作一些非常好的示例可能很有价值。
对于正确的行为空间更广泛、更细微的任务，以更自动化的方式生成大量示例可能很有用，这样可以更有可能为任何运行时输入提供一些高度相关的示例。

**单轮与多轮示例**

在生成示例时需要考虑的另一个维度是示例实际展示的内容。

最简单的示例类型只有用户输入和预期的模型输出。这些是单轮示例。

一个更复杂的示例类型是整个对话，通常模型最初响应不正确，然后用户告诉模型如何纠正其答案。
这被称为多轮示例。多轮示例对于更细致的任务很有用，在这些任务中，展示常见错误并确切说明它们为什么错误以及应该做什么会很有帮助。

## 2. 示例数量

一旦我们有了示例数据集，我们就需要考虑每个提示中应该包含多少示例。
关键的权衡是，更多的示例通常会提高性能，但更大的提示会增加成本和延迟。
而且，超过某个阈值后，过多的示例可能会开始混淆模型。
找到正确的示例数量在很大程度上取决于模型、任务、示例的质量以及您的成本和延迟限制。
据传闻，模型越好，它需要的示例就越少，并且添加更多示例的收益递减速度就越快。
但是，可靠地回答这个问题的最好/唯一方法是进行一些实验，尝试不同的示例数量。

## 3. 选择示例

假设我们不会将整个示例数据集添加到每个提示中，我们需要一种方法根据给定的输入从数据集中选择示例。我们可以这样做：
- 随机选择
- 按输入的（语义或基于关键字的）相似性选择
- 基于其他一些约束，例如令牌大小

LangChain 提供了许多 [`ExampleSelectors`](/docs/concepts/example_selectors)，可以轻松使用任何这些技术。

通常，按语义相似性选择可以带来最佳的模型性能。但这有多重要再次取决于模型和任务，并且是值得尝试的事情。

## 4. 格式化示例

如今，大多数最先进的模型都是对话模型，因此我们将重点关注为它们格式化示例。我们的基本选项是将示例插入：
- 在系统提示中作为字符串
- 作为它们自己的消息

如果我们将其示例作为字符串插入系统提示中，我们将需要确保模型清楚每个示例的开始位置以及哪些部分是输入与输出。不同的模型对不同的语法响应更好，例如 [ChatML](https://learn.microsoft.com/en-us/azure/ai-services/openai/how-to/chat-markup-language)、XML、TypeScript 等。

如果我们将其示例作为消息插入，其中每个示例表示为一系列 Human、AI 消息，我们可能还想为我们的消息分配 [名称](/docs/concepts/messages)，例如 `"example_user"` 和 `"example_assistant"`，以清楚地表明这些消息与最新输入消息的参与者不同。

**格式化工具调用示例**

当示例输出包含工具调用时，将示例格式化为消息是一个棘手的问题。这是因为不同的模型对于在生成任何工具调用时允许的消息序列类型有不同的约束。
- 一些模型要求任何带有工具调用的 AIMessage 必须紧随其后是每个工具调用的 ToolMessages，
- 一些模型还要求任何 ToolMessages 必须紧随 AIMessage 之后，然后才是下一个 HumanMessage，
- 一些模型要求如果聊天历史中有任何工具调用或 ToolMessages，则必须将工具传递给模型。

这些要求是模型特定的，应针对您正在使用的模型进行检查。如果您的模型在工具调用之后需要 ToolMessages 和/或 ToolMessages 之后需要 AIMessages，而您的示例仅包含预期的工具调用而不包含实际的工具输出，您可以尝试在每个示例的末尾添加虚拟的 ToolMessages / AIMessages，并带有通用内容以满足 API 约束。
在这些情况下，尝试将示例作为字符串与消息插入特别值得，因为虚拟消息可能会对某些模型产生不利影响。

您可以在此处 [here](https://blog.langchain.dev/few-shot-prompting-to-improve-tool-calling-performance/) 看到一个案例研究，其中 Anthropic 和 OpenAI 在两个不同的工具调用基准上对不同的少样本提示技术做出响应。
